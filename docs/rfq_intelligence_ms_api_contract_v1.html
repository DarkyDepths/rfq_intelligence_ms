1. Service Identity

A separate, read-heavy analytical microservice. Not a CRUD service — an artifact-producing intelligence engine.

rfq_manager_ms

Owns "today" — RFQ lifecycle, workflows, stages, files, reminders. Publishes domain events.

rfq_intelligence_ms

Owns "memory & insight" — parsing, intelligence artifacts, analytical persistence, read APIs. Reacts to events.

rfq_chatbot_ms

Owns conversation UX only. Routes to manager or intelligence APIs. Produces no intelligence.

Architecture Pattern

Manager publishes
lifecycle events
→
Intelligence reacts
builds artifacts
→
Consumers read
snapshot + artifacts
BACAB flow: routes → controllers → services → datasources for HTTP. event_handlers → services → datasources for async ingestion. Both paths converge at the services layer. Own PostgreSQL database, separate from manager.

Integration Rules

RuleDetail
Database ownershipOwn analytical PostgreSQL DB (rfq_intelligence_db). Never reads from or writes to manager's operational DB.
Manager dependencyThin connector reads only: RFQ id/metadata, file references, workbook reference, minimal display context. Never couples to manager's internal schema.
Event ingestionReacts to 3 events from manager. Sequential build cascades within same handler — not chained event-to-event.
API postureRead-heavy. No CRUD on artifacts — they are built by events, read by consumers. Only reprocess endpoints trigger writes via HTTP.
File storageManager owns files (Azure Blob). Intelligence reads file content via references, never stores raw files.

4. The 6 Intelligence Artifacts

All stored as JSONB payloads in a single artifacts table. Content schema varies by artifact_type.
1 — Partly Deterministic, Partly Heuristic

rfq_intake_profile

Structured extraction from ZIP/MR package. 8 blocks: artifact_meta, source_package, package_structure, canonical_project_profile, document_understanding, field_evidence, quality_and_gaps, downstream_readiness. Foundation artifact — every other output depends on it.

2 — Mixed (Core Active, Historical Future-Only)

intelligence_briefing

Proactive dossier generated on rfq.created. 8 blocks: artifact_meta, briefing_scope, executive_summary, section_a_document_understanding, section_b_similarity_context (unavailable), section_c_preliminary_cost_envelope (unavailable), section_d_compliance_supplier, quality_and_gaps. Dual format: structured fields + summary_text per section.

3 — Deterministic

workbook_profile

Structured extraction from GHI workbook. 8 blocks: artifact_meta, source_workbook, workbook_structure, canonical_estimate_profile (shared mirror), cost_breakdown_profile, financial_profile, schedule_profile, quality_and_gaps. Fixed template = deterministic parsing.

4 — Mostly Deterministic in V1

workbook_review_report

Bridge artifact. 7 blocks: artifact_meta, review_scope, executive_summary, family_1_structural_completeness (active), family_2_benchmark_outlier (dormant), family_3_briefing_vs_actual (selectively active), review_recommendations. Universal finding object shape across all families.

5 — Deterministic Assembly

rfq_intelligence_snapshot

Consumer read model. 8 blocks: snapshot_meta, artifact_refs, availability_matrix, executive_panel, briefing_panel, workbook_panel, review_panel, consumer_hints. One current snapshot per RFQ, rebuilt whenever upstream artifacts change.

6 — Deterministic (Not User-Facing)

rfq_analytical_record

Future analytical store seed. Lifecycle: stub on rfq.created → enriched on workbook.uploaded → enriched on outcome.recorded. Contains normalized features, readiness flags, provenance. Every RFQ enriches the store from its first moment.

Universal Finding Object (workbook_review_report)

{
  "finding_id": "string",
  "family": "structural_completeness | benchmark_outlier | briefing_vs_actual",
  "finding_code": "missing_cost_line.pwht",
  "severity": "low | medium | high",
  "review_posture": "this_deserves_review",
  "reason_plain_language": "PWHT was flagged as required at intake but no PWHT cost line found in workbook.",
  "confidence": "high | medium | low | unavailable",
  "reference_set_summary": { "available": false, "project_count": 0 }
}
Review posture: Always "this deserves review" — never "this is wrong." Intelligence adds a review layer, not a judgment layer.

5. Data Model

One shared table. JSONB content. Versioned artifacts with is_current tracking.

artifacts

Single table for all 6 artifact types — content stored as JSONB, metadata shared.

JSONB rationale: Artifact schemas are deep, nested JSON structures (8 blocks each with sub-objects). Normalizing into relational tables would create dozens of tables. JSONB keeps the DB simple and matches the artifact-first design.

Version Management Rules

is_current invariant: At most one is_current=True row per (rfq_id, artifact_type). Enforced at two levels: (1) datasource layer flips old version's is_current to false and inserts new version in the same transaction; (2) database enforces the invariant through a partial unique index on (rfq_id, artifact_type) where is_current = true.
Two logical semantics, one physical table:
Append-version artifacts (intake_profile, briefing, workbook_profile, review_report, analytical_record): old versions are preserved with is_current=false for traceability. Version history is meaningful — it shows how intelligence evolved for an RFQ.
Replace-current snapshot (rfq_intelligence_snapshot): the current assembled read model for the RFQ. Rebuilt whenever upstream artifacts change. Old snapshot versions have limited historical value — they exist in the table but the business meaning is always "current assembled view." Both use the same table and is_current mechanism in V1.
