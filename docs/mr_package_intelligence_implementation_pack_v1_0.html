What this document defines — the "what"
This is the frozen contract for the MR package parsing lane inside rfq_intelligence_ms. It defines what the parser must produce: the 4-stage pipeline architecture, all frozen @dataclass(frozen=True) contracts, the canonical section taxonomy, deterministic detection rules, spreadsheet extraction contracts, cross-reference checks, the PackageParseEnvelope output shape, and integration points with existing intake_service.py and briefing_service.py. Every structural assumption is validated against the single golden reference package and explicitly flagged as "validated against 1 package."

Convention note: This follows the established repo convention. workbook_parser_implementation_pack_v*.html = the "what" (contracts, schemas, frozen shapes). workbook_parser_blueprint_v*.html = the "how" (files to create, module structure, implementation sequence). The implementation pack defines what the parser must produce; the blueprint defines how it is built. Same convention here.
Single golden reference constraint
This implementation pack is built from one real MR package: SA-AYPP-6-MR-022. Every detection rule, every canonical section mapping, every filename regex is verified against that sample. Rules are intentionally designed to be defensive — they tolerate missing sections, renamed folders, extra files, and unexpected structures. But until a second package validates these patterns, every assumption carries the tag 1-sample.

Validation promotion policy: Any regex, alias, sheet-layout rule, or structural assumption derived from a single sample is tagged validated_against: 1_sample. When a second package confirms the pattern, promote to validated_against: multi_sample. When a pattern breaks against new data, the rule must be revised — not the data forced to fit. This makes every assumption explicitly challengeable by future evidence.

01 Context & Positioning

1.1 Where this fits in the intelligence lifecycle

The rfq_intelligence_ms has two intelligence moments triggered by events from rfq_manager_ms:

Moment 1 — RFQ Arrives (ZIP lane)

Triggered by rfq.created. The MR package (ZIP/folder) is the first real data. This implementation pack defines the parser contract for this moment.

Produces: rfq_intake_profile (enriched with package data), intelligence_briefing (deterministic-enriched v1 — based on structure, filenames, spreadsheets, and cross-checks; not deep document semantics).

Moment 2 — Workbook Uploaded (workbook lane)

Triggered by workbook.uploaded. The GHI estimation workbook is the second data source. Already implemented (Pack 1 + Pack 2).

Produces: workbook_profile, cost_breakdown_profile, parser_report, workbook_review_report.

The two moments are designed to meet: ZIP extraction and workbook extraction share overlapping fields (project name, client, MR number, standards, weight, material types) so the workbook_review_report can compare what the RFQ asked for vs. how GHI priced it.

1.2 Current state of the ZIP lane

intake_service.py currently produces a minimal stub rfq_intake_profile with only manager-provided context (rfq_code, client_name, project_title). The package_structure section says "Source package structure is not parsed yet in this slice." and document_understanding says "Deep document parsing is not implemented."

This implementation pack replaces the stub with real deterministic extraction from the MR package.

1.3 Design philosophy

  • Deterministic before probabilistic. Same principle as the workbook parser. Everything extractable by code comes first. LLM is a future second layer.
  • Package inventory + canonical recognition + targeted extraction + artifact assembly. Four internal stages, but only two persisted artifacts: rfq_intake_profile and intelligence_briefing. The intermediate structures (inventory, registry) are embedded sections inside the artifact, not standalone persisted artifacts.
  • Confidence taxonomy. Every extracted field carries its extraction method: deterministic (derived from structure/patterns), heuristic (strong classification clue, not guaranteed), or deferred (requires LLM or future capability). Never claim certainty where there is inference.
  • Preserve unknowns, never throw them away. Extra folders, unexpected files, unrecognized structures → captured in inventory, exposed as unknown_extra_sections. The parser never pretends it understood everything.

2.3 Input contract

The parser receives a local path to either:

  • A directory (already extracted MR package folder)
  • A ZIP file (extracted to a temporary directory first, then walked)

In cold-start V1, the path comes from ManagerConnector.fetch_package_local_path() — identical pattern to fetch_workbook_local_path(). In production, it will come from Azure Blob Storage download → local temp path.

03 Four-Stage Pipeline

The parser orchestrator executes four sequential stages. Each stage receives the output of the previous one. The pipeline always completes — individual stage failures produce partial results, never crash the whole pipeline.

Stage A — Package Scan
Input: Local filesystem path (directory or ZIP).
Output: PackageInventory — raw tree of every folder and file with normalized paths, sizes, extensions, depth levels.
Method: 100% deterministic. os.walk() + pathlib. ZIP extraction via zipfile when input is a ZIP file.
Failure mode: If the path doesn't exist or ZIP is corrupt → the entire pipeline returns a failed result with error issue. No partial inventory.

Hidden/system files policy: The inventory captures everything, including .DS_Store, Thumbs.db, __MACOSX/, and similar OS artifacts. Each FileEntry carries an is_system_file flag (True for dotfiles, Thumbs.db, __MACOSX contents). Business-level counts (total_files, section file_count) exclude system files. The parser report flags their presence as informational noise, never as warnings or errors.
Stage B — Structure Recognition
Input: PackageInventory.
Output: SectionRegistry — each top-level folder mapped to a canonical section with match method and confidence.
Method: Deterministic number-prefix detection + heuristic name matching for unnumbered folders. See §05 for the full canonical section taxonomy.
Failure mode: Always succeeds. Unrecognized folders → unknown_extra_section. Missing expected sections → noted in missing_sections list.
Stage C — Targeted Extraction
Input: PackageInventory + SectionRegistry.
Output: Per-section extraction results (identity, BOM data, RVL vendors, standards list, SA-175 form IDs, compliance items, etc.).
Method: Deterministic for V1. Each extractor runs independently. A crashed extractor does not block others (same _try_extract pattern as workbook parser).
Failure mode: Per-extractor soft-fail. Three states: None (section not present), synthetic failed result (extractor crashed), normal result (success).
Stage D — Artifact Assembly
Input: All outputs from Stages A, B, C + cross-check results.
Output: PackageParseEnvelope (frozen dict via dataclasses.asdict()).
Method: Pure data assembly. Same pattern as build_envelope() in the workbook parser. Computes parser status from individual stage statuses. Merges all issues, anchor checks, cross-checks.
Failure mode: Always succeeds if Stage A succeeded.

04 Frozen Contracts

All contracts are @dataclass(frozen=True), same as the workbook parser. No mutable state. All fields optional with None defaults unless structurally required.

4.1 Package Inventory contracts

@dataclass(frozen=True)
class FileEntry:
    """Single file in the package."""
    relative_path: str                  # relative to package root
    filename: str                       # basename
    extension: str                      # lowercase, with dot: ".pdf", ".xlsx"
    size_bytes: int
    depth: int                          # 0 = root, 1 = top-level section, 2+ = nested
    parent_folder: str                  # immediate parent folder name
    is_mr_index: bool = False           # True if filename matches MR Index pattern
    mr_number_in_filename: str | None = None   # extracted MR-NNN if present
    section_prefix: str | None = None   # "02" if filename starts with "02_"
    is_system_file: bool = False        # True for .DS_Store, Thumbs.db, __MACOSX contents
    root_role: str | None = None        # "mr_index_root" | "internal_review_file" | "unclassified_root_extra" (only for depth=0 non-system files)


@dataclass(frozen=True)
class FolderEntry:
    """Single folder in the package."""
    relative_path: str                  # relative to package root
    name: str                           # folder basename
    depth: int                          # 0 = root children, 1 = subsections, etc.
    file_count: int
    subfolder_count: int
    number_prefix: int | None = None    # parsed from "NN-Label" pattern
    label: str | None = None            # text after "NN-" prefix


@dataclass(frozen=True)
class PackageInventory:
    """Complete raw inventory of the MR package."""
    package_root_name: str
    input_type: str                     # "directory" or "zip"
    total_files: int                    # business count: excludes system files
    total_files_raw: int                # raw count: includes system files
    total_folders: int
    total_size_bytes: int
    files: list[FileEntry]              # ALL files including system files (raw truth)
    folders: list[FolderEntry]
    root_files: list[FileEntry]         # files at depth 0, excludes system files
    file_extension_counts: dict[str, int]  # business counts: excludes system files
    system_file_count: int              # count of .DS_Store, Thumbs.db, etc.
    scanned_at: str                     # ISO timestamp
Count semantics rule
All count fields in PackageInventory, FolderEntry, and SectionMatch are business counts — they exclude system files (.DS_Store, Thumbs.db, __MACOSX contents). The files[] list contains all files including system files (raw truth). Consumers who need the raw count use total_files_raw or len(files). FolderEntry.file_count is also a business count.

Root-level file classification

Files at depth 0 (directly inside the package root) are classified into three categories:

RootFileRole = Literal[
    "mr_index_root",          # root MR Index.pdf — master table of contents
    "internal_review_file",   # CRS (Comment Review Sheet), internal review spreadsheets
    "unclassified_root_extra" # anything else at root level
]

Detection rules for root files:

  • mr_index_root: filename contains "MR Index" (case-insensitive), no section suffix
  • internal_review_file: filename contains "CRS" or "Internal" (case-insensitive), typically .xlsx
  • unclassified_root_extra: everything else (excluding system files)

The FileEntry for root files should carry a root_role field. In the golden reference: MR Index.pdf → mr_index_root, A.HAK_ Internal CRS-MR-022 REV-00.xlsx → internal_review_file, .DS_Store → system file (excluded).

06 Detection Rules

6.1 Package root identity extraction

The root folder name follows the pattern: {PROJECT-CODE}-MR-{NNN}_{MATERIAL-DESCRIPTION}-REV-{NN}

Regex: r'^([\w-]+-MR-\d+)[_\s]+(.+?)[-\s]*REV[-\s]*(\d+)\s*$'

Example: "SA-AYPP-6-MR-022_COLLECTION VESSEL - CDS-REV-00"
  → mr_number:            "SA-AYPP-6-MR-022"
  → material_description: "COLLECTION VESSEL - CDS"
  → revision:             "00"

Project code extraction from MR number:
  Regex: r'^(.+)-MR-\d+$'
  → project_code:         "SA-AYPP-6"

6.2 Numbered section detection

Pattern: folder name starts with "NN-" where NN is 0-99
Regex: r'^(\d{1,2})\s*[-–]\s*(.+)$'

Example: "06-Applicable Standards"
  → number_prefix: 6
  → label:         "Applicable Standards"

Match confidence: "high" (number prefix is unambiguous)

6.3 MR Index file detection

Pattern: filename matches "MR Index" (case-insensitive) optionally followed by "_NN"
Regex: r'^MR\s*Index(?:[_\s]*(\d+))?\.\w+$'  (case-insensitive)

Examples:
  "MR Index.pdf"      → root MR Index, section_number=None
  "MR Index_06.pdf"   → section 06 MR Index
  "mr index_03.pdf"   → section 03 MR Index (case variation)

6.4 Content file prefix detection

Pattern: filename starts with "NN_" matching its parent section number
Regex: r'^(\d{1,2})[_\-]'

Example in folder "02-MR Description-BOM":
  "02_BOM_SA-AYPP-6-MR-022_COLLECTION VESSEL- CDS.xlsx" → prefix=02 ✓ matches section
  "02_Scope Of Supply.pdf" → prefix=02 ✓ matches section

Cross-check: prefix should match parent section number. Mismatch → warning issue.

6.5 MR number cross-reference detection

Scan all filenames for MR-NNN pattern:
  Regex: r'MR-(\d+)' across all file names

Golden reference findings:
  MR-022 → found in 6 files (matches package root)
  MR-023 → found in 2 files (03-RVL/SA-AYPP-MR-023_ RVL-.docx and .pdf)

Rule: any MR number != primary → added to mr_number_mismatches → cross-check warning.
  This is a real review flag: the RVL might reference a different MR.

6.6 Standards filename patterns

All patterns are case-insensitive:

SAMSS:    r'SAMSS[-_]?(\d{3})'           → "SAMSS-010", "01-SAMSS-043"
SAES:     r'SAES[-_]?([A-Z][-_]\d{3})'   → "SAES-A-004", "SAES-W-010"
SAEP:     r'SAEP[-_]?(\d+)'              → "SAEP-1142", "SAEP-31"
SA-175:   r'175[-_](\d{6})'              → "175-010710", "175-321900"

Numeric prefix stripping for SAMSS: "01-SAMSS-043.PDF" → strip leading "01-" → "SAMSS-043"

6.7 Standards subfolder detection

Folder 06 may contain numbered subfolders:
  Pattern: r'^(\d+)\.\s*(.+)$'
  Examples: "1. SAMSS" → (1, "SAMSS"), "4. STD DWG" → (4, "STD DWG")

  Canonical subfolder map:
    contains "samss" → family "samss"
    contains "saes"  → family "saes"
    contains "saep"  → family "saep"

    otherwise        → family "other"

10 Integration with Existing Code

10.1 intake_service.py changes

The current build_intake_profile_from_rfq_created() method produces a stub. After this blueprint is implemented:

  • When a package path is available: call PackageParserOrchestrator.parse() → build enriched artifact
  • When no package path is available (manager doesn't provide one yet): fall back to current stub behavior
  • The method signature doesn't change — it still receives rfq_context and event_meta

10.2 briefing_service.py changes

The current build_briefing_from_intake() reads intake content and produces a stub briefing. After this blueprint, the briefing becomes a deterministic-enriched v1 — populated from structure, filenames, spreadsheet data, and cross-checks. It does not yet include semantic document understanding (QAQC content, specs analysis, etc. — those require LLM and are deferred).

  • When intake has "status": "parsed" in package_structure: enrich briefing with deterministic data (standards count, vendor list summary, BOM summary, review flags from cross-checks)
  • When intake has "status": "limited": keep current stub behavior
  • The briefing's what_is_known section expands to include package-derived fields
  • The briefing's compliance_flags_or_placeholders section activates with real standards data

10.3 ManagerConnector changes

Add fetch_package_local_path(package_ref: str) -> Path — same pattern as fetch_workbook_local_path(). In cold-start V1, resolves to a local fixtures directory.

10.4 app_context.py changes

Add get_package_parser_orchestrator() -> PackageParserOrchestrator provider. Wire into get_intake_service() if direct injection is needed, or keep it as a direct instantiation inside intake_service (same pattern as workbook_service calling parse_workbook_deterministic()).

10.5 Dependencies

New Python dependencies required:

  • openpyxl — for reading .xlsx files (BOM, Compliance, Deviation List, SDS)
  • python-docx — for reading RVL .docx files

Both are pure Python, no system dependencies. Add to requirements.txt.
