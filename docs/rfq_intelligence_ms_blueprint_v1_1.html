Step 01 — Mission & Boundaries

Mission, Ownership & Cold-Start Scope

Defining what rfq_intelligence_ms is, what it owns, what it must not own, and what is realistically feasible in V1 with one real RFQ package and one real workbook.

Validated Mission Statement
rfq_intelligence_ms is a separate, read-heavy analytical microservice that transforms RFQ source material and historical records into persistent intelligence artifacts for multiple consumers.
It does not manage lifecycle state, does not send actions, does not own conversation, and does not mutate the operational backbone owned by rfq_manager_ms. Intelligence produces insight; other services decide what to do with it.

What It Owns

Document / RFQ parsing
Historical matching
Cost-range support logic
Supplier / client / historical intelligence
KPI and analytical computation
Its own historical / analytical database
Persistent intelligence artifacts

What It Must Not Own

Workflows, stages, or stage transitions
Reminders, communication, notifications
Chatbot sessions, intent detection, conversation history
IAM / auth
File storage
Workflow execution decisions
UI-specific presentation logic

Ownership Split

rfq_manager_ms

Owns "Today" — Operational Truth

RFQ lifecycle state, workflows, stages, subtasks, files, reminders. The operational core and stable backend foundation.

rfq_intelligence_ms

Owns "Memory & Insight" — Analytical Truth

Parsing, extraction, historical matching, cost analysis, intelligence artifacts. The shared analytical brain of the platform.

Cold-Start Reality

Current data: One real corresponding GHI estimation workbook (fixed template reference). This is the "golden reference pair." The intelligence engine starts empty and capabilities must be released in phases matching available data.

V1 is a document-and-workbook intelligence service first, and a historical analytics service later.

Layer A — Build Now, User-Facing

RFQ intake parsing. Briefing generation. Workbook parsing. ZIP ↔ workbook comparison. Structural/completeness checks based on template logic. Persistence of intelligence artifacts and explanations.

Layer B — Scaffold Now, Dormant

Similarity engine interfaces. Benchmark engine interfaces. Confidence model. Historical tables for outcomes/comparables. Placeholders for cost-envelope and anomaly families needing history. Explicitly marked "insufficient historical base."

V1 Check Classification

ZIP/package convention checksHeuristicReliable only where the package follows known Aramco MR structure; otherwise clearly marked

Demo & Mock Data Policy

Golden reference pair: The 1 real RFQ + 1 real workbook is the validation anchor. All design and parsing logic is validated against this pair first.

Synthetic fixtures: Allowed for screen states, API contract testing, UI demos, and test scenarios. Clearly labeled and stored separately.

Hard rule: Never mix synthetic fixtures into claimed historical intelligence. Never use mock data to claim realistic benchmarks, win probabilities, or trustworthy cost ranges.
Validated Conclusion: rfq_intelligence_ms V1 is a cold-start intelligence service centered on extraction, briefing, workbook understanding, and explainable comparison between "what was requested" vs "how GHI prepared the estimate." Historical benchmarking and predictive outputs remain scaffolded but not active until real data arrives.
Step 02 — Artifact Design

V1 Artifact Set — The Six Artifacts

The exact outputs rfq_intelligence_ms persists and serves for a single RFQ. All proactive intelligence outputs are first-class persistent artifacts with status, timestamp, and version.

Artifact 1 — Partly Deterministic, Partly Heuristic

rfq_intake_profile

Structured extraction artifact from the ZIP/MR package. Normalizes the incoming RFQ into one machine-readable profile: folder structure, detected document families, extracted standards, vessel/project metadata, material/tonnage/dimensions, RVL/mandated vendor indicators, scope/QA-QC fields, and extraction provenance with confidence. This is the foundation — every other intelligence output depends on it, not on raw files.

Artifact 2 — Mixed (Core Sections Active, Historical Sections Future-Only)

intelligence_briefing

The flagship proactive output. Generated automatically when the RFQ enters the system, before the first human opens the RFQ. Stored as a first-class dossier with status, timestamps, confidence per section, and version. In cold start, active sections include document understanding and compliance/risk flags; similarity and cost envelope sections are present but marked unavailable.

Artifact 3 — Deterministic

workbook_profile

Structured extraction artifact from the GHI estimation workbook. Contains workbook metadata, template recognition, parsed key sheets, item tables, summary totals, cost breakdown, and extracted comparable fields shared with intake profile. The cleanest V1 artifact because the workbook template is fixed and deterministic to parse.

Artifact 4 — Mostly Deterministic in V1

workbook_review_report

The first real bridge between intake and workbook intelligence. Contains three anomaly families: structural/completeness (fully active), briefing-vs-actual (selectively active), and benchmark/outlier (schema-ready but suppressed). Every finding phrased as "this deserves review," never "this is wrong."

Artifact 5 — Deterministic Assembly

rfq_intelligence_snapshot

Consumer-facing read model. A single assembled projection per RFQ containing latest states of all intelligence artifacts, section availability, confidence markers, and consumer hints. Simplifies UI and chatbot integration — consumers read the snapshot rather than stitching internal artifacts.

Artifact 6 — Deterministic

rfq_analytical_record

Not user-facing. The seed for the future analytical store. Lifecycle: created as a stub on rfq.created with intake-extracted normalized features and readiness flags; enriched on workbook.uploaded with workbook normalized features and cost metrics; enriched again on outcome.recorded with result/outcome fields and provenance. Every new RFQ enriches this store from its first moment in the system, so future matching and benchmarking improve over time.

What Is Explicitly Future-Only

True similarity_matches
True cost_envelope_report
True client_history_report
True supplier_intelligence_report
True benchmark_outlier_report
Bid strategy / win probability artifacts

These all appear in the target architecture but require historical data maturity. In V1 they exist as empty sections, "not enough historical data" markers, or synthetic demo fixtures clearly separated from real intelligence.

Step 05 — Normalization Contract

Normalization Rules & Null Semantics

The rules that make the intake side and workbook side actually meet. Every canonical field carries one normalized value for comparison; evidence and ambiguity stay outside.

The 10 Normalization Rules

Rule 1 — 4-State Null Semantics

known unknown not_applicable ambiguous

For must-have comparison fields (project_family, equipment_type, material_family, material_grade_primary, weight_profile, process_flags), use explicit state tracking: {"value": "...", "state": "known | unknown | not_applicable | ambiguous"}. For lighter fields, nullable values are sufficient.

Why this matters: unknown means "maybe later with better docs." not_applicable means "should not be expected." ambiguous means "needs review." Plain null loses business meaning.

Rule 2 — Controlled Vocabulary Before Free Text

If a field is used for comparison, similarity, filtering, or workbook matching, it must use a controlled vocabulary. The normalization pipeline is always: raw extraction → synonym mapping → canonical token.

Rule 3 — Raw Text Only as Fallback

Keep structured normalized subfields when possible. Keep one fallback normalized text field. Never compare long raw prose directly.

Rule 4 — Units Canonicalized at Write Time

DimensionCanonical Unit
Dimensionsmm
Weightkg
Durationvalue + unit (day|week|month) — comparison code converts consistently
Quantitynumeric + explicit unit
DatesISO date

Store canonical numeric value in the profile; original unit preserved in field_evidence only. Downstream comparison code never converts units on the fly. Exception — duration: stored as value + unit (day|week|month) in the canonical profile, not forced to days. Comparison code handles conversion consistently. This avoids lossy conversion where "1 month" and "30 days" carry different business signals.

Rule 5 — Multi-Value Handling

For array fields (applicable_standards, material_grade_secondary, mandated_vendors, qa_qc_requirements, scope_tags): deduplicate, sort deterministically, canonicalize synonyms, preserve order only if business meaning depends on it.

Rule 6 — Process Flags Use Tri-State or Richer

RT/PWHT/U-Stamp/NB never use plain boolean. Use explicit categorical states: full | spot | none | unknown | ambiguous for RT, yes | no | unknown | ambiguous for others.

Rule 7 — Primary vs Secondary Value Split

For fields where one dominant value matters for comparison (especially material grade), split primary from secondary. The future similarity engine needs a dominant comparable value first.

Rule 8 — Never Normalize Into Unsupported Business Claims

No cost-per-ton, pricing assumptions, benchmark class, win probability, or "complexity score" in the intake canonical profile. Those are intelligence outputs requiring later stages or historical data.

Rule 9 — Shared Enums Across Intake and Workbook

Non-negotiable shared enums: project_family, material_family, process_flags.*, quantity basis, weight basis, delivery duration unit — all must be defined once and reused by both extractors. Intake and workbook must never invent separate taxonomies. This is the core guarantee behind the overlapping extraction templates.

Shared controlled-open vocabularies: project_subtype, equipment_type — use seed lists with alias dictionaries, extensible without schema change.

Rule 10 — Confidence Outside the Canonical Value

Do not store confidence inside the canonical comparison block. Canonical profile stores the normalized value. field_evidence stores extraction method and confidence. quality_and_gaps stores whether the field is ambiguous or weak. This keeps the comparable surface clean.

Step 07 — Mapping Logic

Raw → Canonical Mapping Rules

The pipeline that makes the schema executable: raw evidence → extractor output → canonical mapper → canonical field. Deterministic wherever possible, conservative when uncertain, shared across intake and workbook for overlapping fields.

5 Global Mapping Rules

Rule A — Deterministic Beats Inferred

Workbook fixed cells, folder position, BOM columns, standards filenames outrank any LLM interpretation.

Rule B — One Mapper, Many Extractors

Intake and workbook extractors can differ, but they must feed the same canonical mapper for shared fields. Prevents one side writing "horizontal_vessel" and the other "H vessel."

Rule C — No Forced Guessing

If the mapper cannot safely choose, output unknown or ambiguous. Never manufacture a clean value for UI aesthetics.

Rule D — Value First, Evidence Elsewhere

The mapper outputs only normalized values. Why it picked them belongs in field_evidence.

Rule E — Precedence Table per Field

Each must-have field has a ranked source order. Higher-ranked sources override lower-ranked ones. This avoids chaotic merging.

Field-by-Field Mapping

project_family — Source Precedence

1) Workbook item/general sheet → 2) BOM/item descriptions → 3) Specs/datasheets titles → 4) Drawing titles → 5) LLM from prose. Uses synonym dictionary. Conflict: two strong different families with no dominant → unknown + ambiguous.

material_family — Grade-Pattern Inference

SA-516, A516, CS → carbon_steel. SS304, 316L → stainless_steel. 2205, Duplex → duplex. Inconel, 625, 825 → nickel_alloy. Multiple families across major components → mixed.

weight_profile

Parse number + source unit → convert to kg → set basis. If only a range, do not fake a point value. If both per-item and total weight exist, canonical prefers total_project. Never estimate weight from dimensions in V1 — that is engineering inference, not extraction.

applicable_standards

1) Folder 06 filenames (highest confidence) → 2) Exact mentions in specs/docs → 3) LLM from prose. Normalize variants (ASME Sec VIII Div.1 → ASME VIII Div 1). Deduplicate. If both broad and precise forms exist, keep precise and drop broad.

Cross-Field Inference Rules

RuleDirectionSafety
material_family from material_grade_primaryGrade → familySafe and useful
project_family from equipment_typeEquipment → familySafe with dictionary guard

Cost fields from intake technical dataBLOCKEDNever in V1. No converting material + weight + PWHT into price-like facts.

Step 08 — End-to-End Resolution

process_flags — Candidate Resolution

One of the strongest shared comparison families. RT, PWHT, U-Stamp, and NB are explicitly named as major cost drivers and overlapping fields between ZIP and workbook.

Candidate Object Shape

{ "field": "pwht_required", "candidate_value": "yes", "source_kind": "checklist | datasheet | spec_text | llm_text | workbook_cell", "source_ref": "file/page/cell reference", "strength": "high | medium | low" }

Strength Ranking

StrengthSources
highWorkbook fixed cell, structured checklist, explicit table field
mediumExact sentence in structured doc/spec
lowLLM-derived interpretation from prose or indirect wording

Resolution Policy (V1)

Phase 1: Normalize all raw expressions into canonical candidate values. Phase 2: Rank by source strength. Phase 3: One strongest candidate → choose it. Several strongest with same value → choose it. Several strongest with conflicting values → ambiguous. No usable candidate → unknown.

Step 09 — End-to-End Resolution

applicable_standards — Candidate Resolution

One of the cleanest V1 assets. Standards auto-detection from Folder 06 filenames is a low-complexity, cold-start-friendly capability.

Standard Classification

TypeExamplesV1 Treatment
Exact coded standardsSAMSS-010, SAES-A-004, SAEP-302Best targets — filenames expose them deterministically
Named code familiesASME VIII Div 1, API 650, ASTM A240From filenames, datasheets, specs, workbook notes
Broad references (too vague alone)ASME, API, ASTMOnly stored if no precise variant exists

Resolution Logic (Multi-Value)

Phase 1: Normalize all candidates into canonical code/display forms. Phase 2: Union across all sources. Phase 3: Collapse broad duplicates — if both ASME and ASME VIII Div 1 exist, keep precise, drop broad. Phase 4: Uncertain/truncated codes → do not inject into canonical list, store in field_evidence.

What NOT to Do

Do not parse full PDF standard contents in V1. Do not assume every Folder 06 file is equally applicable. Do not infer process flags from standards presence. Do not infer cost impact from standards alone. Standard detected ≠ PWHT=yes. Standard detected ≠ cost uplift=X.

The two extraction templates are designed to meet. The workbook mirror uses the exact same field names, enums, and normalization rules as intake on overlapping fields.

Step 12 — Cost Intelligence

cost_breakdown_profile — Field-by-Field Schema

The most important workbook-native block. Represents estimate cost structure — not revenue/margin strategy, not payment risk, not schedule.

7 Sub-Blocks

1) currency_context

Base currency: SAR. Secondary: USD. Exchange rate if extractable. All canonical totals stored in SAR; USD as optional mirror.

2) headline_totals

total_estimated_cost_sar, total_estimated_cost_usd, total_cost_per_ton_sar, total_cost_per_ton_usd, total_weight_kg_reference, cost_basis_state. No margin or revenue — those belong in financial_profile.

3) cost_lines — V1 Line Codes

materiallabourdesignconsumablesndtpwhtpaintu_stampnbtransportfinancial_chargesoverheadscontingencysubcontractingother

Each line: line_code, label, amount_sar, amount_usd, share_pct, cost_per_ton_sar, cost_per_ton_usd, source_sheet, state.

4) cost_shares

Compact summary: material_pct, labour_pct, design_pct, ndt_pct, pwht_pct, paint_pct, financial_pct, overheads_pct, contingency_pct, other_pct. Pre-computed for fast comparison — avoids re-aggregating the line array.

5) unit_economics

cost_per_ton_sar, cost_per_ton_usd, material_cost_per_kg_sar, material_cost_per_kg_usd, derived_from_weight_kg. These stay workbook-native, not promoted into the shared mirror.

6) supporting_cost_drivers

Bridge block: rt_requirement, pwht_required, u_stamp_required, nb_registration_required, item_count, total_weight_kg, material_family, material_grade_primary. Values always copied from canonical_estimate_profile — never independently extracted.

7) quality_and_gaps

missing_expected_cost_lines, zero_value_suspicious_lines, inconsistent_total_vs_line_sum, missing_cost_per_ton, missing_weight_reference, sheet_parse_warnings.

Key Design Decisions

Labour: Headline cost line in V1. Optionally split into direct_labour/indirect_labour if extraction is stable.

Financial charges: Headline cost line in cost_breakdown_profile + detailed decomposition (LC, bank guarantee, performance bond, advance payment, negative cash flow) in financial_profile. Avoids duplication while preserving both views.
Step 13 — Bridge Artifact

workbook_review_report — The Three Families

The first real bridge between intake intelligence and workbook intelligence. Review-oriented, not parser or decision artifact. Phrased as "this deserves review," never "the workbook is wrong."

Family Activation in Cold Start

Family 1 — Structural/Completeness

Fully Active

Missing sheets, missing cost lines, line-total mismatch, broken percentage distribution, missing weight reference, template mismatch. The safest and most detailed V1 section. Also exposes structural_score and parse/comparison blockers.

Family 2 — Benchmark/Outlier

Schema-Ready, Dormant

status: "unavailable", activation_reason: "insufficient_historical_base". Cost-per-ton, share, margin, schedule comparisons require historical peers. The honest V1 position.

Family 3 — Briefing vs Actual

Selectively Active

Shared field deviation: intake says carbon_steel, workbook says stainless_steel. Requirement coverage gap: intake flags PWHT=yes, no PWHT cost line in workbook. Cost envelope gap: inactive in cold start unless briefing supports it. This family is the strongest V1 demonstration output.

Critical rule: Family separation must be preserved visibly in the report. Do not flatten structural facts and benchmark facts into one priority list — they have different trustworthiness levels. Users must see which family each finding comes from.

Every section has both: a summary_text (for managers and chatbot) and structured fields underneath (for machines and stable UI). The briefing must be queryable by multiple consumers — UI panels, chatbot, and dashboards. Pure prose is hard to reuse; pure JSON is weak for human review.

Step 16 — Integration & API

Integration Flow, Events, API & Persistence

The step that turns the artifact model into an actual microservice boundary. Event-driven ingestion, synchronous read API, own database, clean ownership boundaries.

Ownership Boundary

rfq_manager_ms

RFQ lifecycle state, workflow/stages, file upload/storage, outcome capture, operational truth.

rfq_intelligence_ms

Parsing triggered by events, intelligence artifacts, analytical projections, historical enrichment, read APIs.

rfq_chatbot_ms

Conversation UX only. Routes to manager or intelligence APIs. Produces no intelligence.

V1 Event Triggers

EventSourceIntelligence Reaction
rfq.createdManagerFetch metadata → build rfq_intake_profile → build intelligence_briefing → create rfq_analytical_record stub from intake features → update snapshot
workbook.uploadedManagerFetch workbook ref → build workbook_profile → build workbook_review_report → enrich rfq_analytical_record with workbook features → update snapshot

Only 3 events in V1. Do not introduce a large event catalog yet. These 3 cover the complete intelligence lifecycle. Each event triggers a sequential build chain within the same handler — not separate events triggering each other (avoids cascades and race conditions).

Persistence Strategy

Own database: Intelligence has its own analytical PostgreSQL database, separate from manager's operational DB.

Artifact versioning: Append-version artifacts (immutable, traceable) + replace-current snapshot (latest assembled view). When a new version is created, the old version's is_current flips to false atomically in one transaction.

Common metadata per artifact: artifact_id, rfq_id, artifact_type, version, status, created_at, source_event_type, source_event_id, is_current.

Snapshot: One current snapshot per RFQ, regenerated whenever upstream intelligence changes.

Historical store: rfq_analytical_record grows organically — no retraining needed. Seed for future similarity and benchmarking.

Reprocessing Cascade Rules

TriggerRebuilds
Source package references changedintake_profile → briefing → analytical_record (re-stub) → snapshot
New workbook version uploadedworkbook_profile → review_report → analytical_record (enrich) → snapshot
Intake profile version changesbriefing → snapshot
Workbook profile or briefing changesreview_report → snapshot

All cascades run as sequential steps within the same event handler — never as separate events triggering each other.

What Intelligence Reads from Manager

Only: RFQ id/metadata, file references, workbook reference, minimal operational context for display. Intelligence should not become dependent on manager's internal operational schema. Build once on event, read many times afterward — that is the right pattern.

Failure & Partial-Completion Model

Both artifact-level and snapshot-level support: pending, partial, complete, failed. One failed section should not hide the whole RFQ intelligence state. The UI renders progressively as sections complete.

Complete V1 Flow Summary
Events → Artifacts → Snapshot → Consumers
Manager publishes lifecycle facts. Intelligence reacts asynchronously and builds versioned analytical artifacts plus a current per-RFQ snapshot. UI and chatbot read that snapshot and underlying artifacts via synchronous request-response APIs. Intelligence never writes to the operational database. The chatbot stays thin. Each service owns its own database.
