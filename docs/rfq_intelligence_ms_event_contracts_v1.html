rfq_intelligence_ms — Event Contracts

Asynchronous event contracts consumed by the shared intelligence engine — V1 cold-start scope.
Version: 1.0 Date: March 2026 Events: 3 Producer: rfq_manager_ms Consumer: rfq_intelligence_ms Status: V1 Contract — Frozen

1. Scope

What this document covers and what is documented elsewhere.

This document defines the asynchronous event contracts consumed by rfq_intelligence_ms. It specifies the 3 V1 lifecycle events that trigger intelligence artifact creation and refresh.

HTTP endpoints are documented separately in the API Contract and OpenAPI YAML. These events are not modeled as REST endpoints — they represent a separate inbound adapter in the BACAB architecture.

Relationship: Events trigger artifact builds. HTTP endpoints serve the resulting artifacts to consumers. The two contracts are complementary but independent: changing an event payload does not change the HTTP response shape, and vice versa.

rfq_manager_ms is the producer. It emits domain events at key lifecycle moments: when an RFQ is registered, when a workbook is uploaded, and when an outcome is recorded. rfq_intelligence_ms is the consumer. It reacts to these events by building or refreshing persistent intelligence artifacts in its own analytical database.

Each event triggers a sequential build chain inside the same handler execution. Artifact creation steps happen in order within the handler — they are not dispatched as additional events. This avoids cascading event loops and race conditions.

After any artifact build chain completes, the rfq_intelligence_snapshot is reassembled to reflect the latest intelligence state for the RFQ. Consumers (UI, chatbot, dashboards) read the snapshot and underlying artifacts via the HTTP API.

BACAB adapter separation: Event handlers are a separate inbound adapter from HTTP routes. Both converge at the services layer.
HTTP path: routes → controllers → services → datasources
Event path: event_handlers → services → datasources

3. Standard Event Envelope

Common fields present on every V1 event. All fields use snake_case.
FieldTypeRequiredDescription
event_idUUID (string)RequiredGlobally unique identifier for this event instance. Used for idempotency and deduplication.
event_typeStringRequiredDiscriminator: rfq.created, workbook.uploaded, or outcome.recorded.
event_versionStringRequiredSchema version of this event type. V1 = "1.0". Must be explicit so consumers can detect breaking changes.
producerStringRequiredIdentifier of the emitting service. V1 = "rfq_manager_ms".
emitted_atISO 8601 datetimeRequiredTimestamp when the producer emitted this event.
payloadObjectRequiredEvent-type-specific data. Always contains rfq_id. Additional fields vary by event type.
Envelope shape
{
  "event_id":      "uuid",
  "event_type":    "rfq.created | workbook.uploaded | outcome.recorded",
  "event_version": "1.0",
  "producer":      "rfq_manager_ms",
  "emitted_at":    "ISO 8601",
  "payload": {
    "rfq_id":      "uuid",
    // ... event-specific fields
  }
}
Payload contains references, not file content. Events communicate metadata and storage references (blob paths, filenames, timestamps). File binary content is never embedded in event payloads. Intelligence fetches file content separately via the manager connector.

4. Event Contracts

One section per V1 event. Each defines trigger meaning, payload, processing expectation, and artifact outcomes.
EVENT rfq.created producer: rfq_manager_ms

Trigger: Emitted when a new RFQ is registered in rfq_manager_ms and its initial source-package references (ZIP/MR folder) are available. This is the first intelligence moment — it wakes up the intelligence engine.

Purpose: Provides the minimal information needed for intelligence to start intake parsing and briefing generation. Intelligence fetches full RFQ metadata and file references from the manager connector — the event itself carries only the RFQ identifier.

EVENT workbook.uploaded producer: rfq_manager_ms

Trigger: Emitted when an estimation workbook is uploaded or replaced for an RFQ in rfq_manager_ms. This is the second intelligence moment — it produces the workbook analysis and the intake-vs-workbook bridge.

Purpose: Provides the workbook storage reference so intelligence can fetch and parse it deterministically. The workbook template is fixed (36-sheet GHI format), making parsing a deterministic engineering problem.

EVENT outcome.recorded producer: rfq_manager_ms

Trigger: Emitted when the final outcome of an RFQ is recorded in rfq_manager_ms — awarded, lost, or cancelled. This closes the feedback loop and enriches the analytical store for future intelligence.

Purpose: Provides the outcome status so intelligence can record it alongside the already-extracted intake and workbook features. This data point is critical for future similarity weight calibration, win-probability modeling, and benchmark ranges — but those capabilities are dormant in V1 cold start.

Timestamp note: emitted_at is the event/message timestamp; recorded_at is the business/domain timestamp when the outcome was recorded in the producer system.

6. Contract Rules & Invariants

V1 rules that both producer and consumer must respect.
  1. All envelope fields (event_id, event_type, event_version, producer, emitted_at, payload) are mandatory on every event.
  2. Payload fields marked "Required" in each event contract must be present. Optional fields may be absent or null.
  3. Unknown extra fields in the payload must not break the consumer. Intelligence ignores fields it does not recognize (forward compatibility).
  4. Duplicate delivery may happen. The consumer must be idempotent — it checks event_id against source_event_id on existing artifacts before processing.
  5. Events communicate references (IDs, blob paths, metadata), never raw file content or binary data.
  6. event_version must be explicit on every event. If the producer changes the payload schema in a breaking way, it must increment event_version.
  7. The producer (rfq_manager_ms) owns lifecycle truth. Intelligence derives analytical artifacts only — it never writes back to the operational database or modifies RFQ lifecycle state.
  8. Build chains are sequential within the same handler execution. Intelligence must not emit additional events to trigger its own downstream steps (no event chaining).
  9. If any step in a build chain fails, the handler must record partial status on artifacts already created and not silently discard work. The snapshot should reflect partial availability.

7. Out of Scope for V1

What is explicitly not part of these event contracts.
Historical benchmarking payloads in events
Similarity outputs or match results in events
Win probability or bid strategy payloads
Manager internal DB schema leakage in payloads
File binary content embedded in event payloads
Intelligence-to-intelligence chained lifecycle events
Events emitted by rfq_intelligence_ms itself
Chatbot or UI events in this contract

9. Implementation Note

Relationship between this document and other contract artifacts.

This document is the frozen human-readable asynchronous contract for V1. It defines the business meaning, payload shapes, processing expectations, and invariants for the 3 lifecycle events consumed by rfq_intelligence_ms.

The HTTP API is documented separately in the API Contract and the OpenAPI YAML. Together, these three documents form the complete V1 interface contract for rfq_intelligence_ms.

Machine-readable event schemas (e.g., AsyncAPI, JSON Schema) can be produced later from this document without changing the business meaning. The human-readable contract takes precedence if any machine-generated schema deviates.
